# 04-01: Version Bump + Release Workflow

## Goal

Prepare `@kindlm/core` and `@kindlm/cli` for their synchronized v1.0.0 release by:
1. Creating a major changeset that bumps both packages together
2. Adding `NPM_CONFIG_PROVENANCE=true` and the required `id-token` permission to `release.yml`

No external service dependencies. All changes are local file edits.

## Context

- `@kindlm/core` is at v0.2.1; `@kindlm/cli` is at v0.4.1
- `.changeset/config.json` already links the two packages — a single major changeset will bump both to 1.0.0
- `release.yml` uses `changesets/action@v1` but is missing provenance support

## Requirements Covered

- REL-01: Synchronized versions via Changesets (core + cli → 1.0.0)
- REL-02: npm provenance enabled (`NPM_CONFIG_PROVENANCE=true`)
- REL-03: GitHub Release created with changelog (changesets/action handles this automatically)

## Wave

Wave 1 — no dependencies on other plans

---

## Tasks

### Task 1 — Create the v1.0.0 changeset file

Create the file `.changeset/stable-release.md` with the following exact content:

```
---
"@kindlm/core": major
"@kindlm/cli": major
---

Initial stable release of KindLM v1.0.0
```

**File path:** `.changeset/stable-release.md`

**Why this works:** Changesets reads every `.md` file in `.changeset/` (except `README.md`). Because `linked` in config.json groups `@kindlm/core` and `@kindlm/cli`, a `major` bump on either one will force both to the same version. Starting from 0.x, a major bump produces 1.0.0 for both.

**Expected outcome after `changeset version` runs:**
- `packages/core/package.json` → `"version": "1.0.0"`
- `packages/cli/package.json` → `"version": "1.0.0"`
- `CHANGELOG.md` generated in each package

---

### Task 2 — Add `id-token` permission to `release.yml`

npm provenance requires the workflow to request an OIDC token. Add `id-token: write` to the `permissions` block.

**File:** `.github/workflows/release.yml`

Change the `permissions` block from:

```yaml
permissions:
  contents: write
  pull-requests: write
```

To:

```yaml
permissions:
  contents: write
  pull-requests: write
  id-token: write
```

---

### Task 3 — Add `NPM_CONFIG_PROVENANCE=true` to the publish step

**File:** `.github/workflows/release.yml`

In the `Create Release PR or Publish` step, add `NPM_CONFIG_PROVENANCE: "true"` to the `env` block:

```yaml
      - name: Create Release PR or Publish
        uses: changesets/action@v1
        with:
          publish: npx changeset publish
          title: "Version Packages"
          commit: "chore: version packages"
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
          NPM_CONFIG_PROVENANCE: "true"
```

**Why `"true"` as a string:** The npm config env var is parsed as a string; quoting avoids YAML boolean coercion issues.

---

### Task 4 — Commit and push to trigger the release PR

Run these commands from the repo root:

```bash
git add .changeset/stable-release.md .github/workflows/release.yml
git commit -m "chore: add v1.0.0 changeset and enable npm provenance"
git push origin main
```

**Expected outcome:** The `release.yml` workflow runs. Because a changeset is present, `changesets/action` opens a "Version Packages" pull request updating both `package.json` files and generating `CHANGELOG.md` entries.

---

### Task 5 — Merge the Version Packages PR

Once the PR is open:
1. Review the PR — verify both packages show `1.0.0` in the diff
2. Merge the PR into `main`
3. `release.yml` runs again — this time publishes to npm and creates a GitHub Release

**Expected outcome:**
- `npm view @kindlm/core version` → `1.0.0`
- `npm view @kindlm/cli version` → `1.0.0`
- `npx @kindlm/cli@latest --version` → `1.0.0`
- npm provenance attestation visible on npmjs.com package pages
- GitHub Release exists at `https://github.com/<org>/kindlm/releases/tag/@kindlm/core@1.0.0`

## Verification

```bash
# Verify changeset exists
ls .changeset/stable-release.md

# Verify workflow has provenance env var
grep -n "NPM_CONFIG_PROVENANCE" .github/workflows/release.yml

# Verify workflow has id-token permission
grep -n "id-token" .github/workflows/release.yml

# After publish — verify npm versions
npm view @kindlm/core version
npm view @kindlm/cli version
npx @kindlm/cli@latest --version
```

Expected grep output:
```
  id-token: write
  NPM_CONFIG_PROVENANCE: "true"
```

---

## Manual Checkpoint: npm Publish

The actual npm publish happens when `changesets/action` runs after the Version Packages PR is merged. This requires `NPM_TOKEN` to be set as a GitHub Actions secret. Verify before merging:

- [ ] `NPM_TOKEN` secret exists in GitHub repo settings → Settings → Secrets → Actions
- [ ] Token has publish permissions for the `@kindlm` npm org scope

If the token is missing or expired, the publish step will fail with a 403. Set/refresh the token before merging.
