# 04-02: Monitoring + E2E Verification

## Goal

Verify the health endpoint, configure uptime monitoring for `api.kindlm.com/health`, and run a full E2E smoke test to confirm the production release is working end-to-end.

- MON-01 (health endpoint) is already implemented in `packages/cloud/src/index.ts` — verify and document only

- E2E smoke tests exist at `packages/cli/tests/e2e/smoke.test.ts`

- MON-02: UptimeRobot monitoring `api.kindlm.com` every 5 minutes

## Tasks

### Task 1 — Verify the health endpoint in production

Run from any terminal with curl:

```bash
curl -s https://api.kindlm.com/health
```

**Expected response:**
```json
{"status":"ok"}
```

**Expected HTTP status:** 200

If the response is not 200 or the body does not contain `"status":"ok"`, investigate `packages/cloud/src/index.ts` and redeploy via:

```bash
npx wrangler deploy --env production
```

The implementation is in `/Users/petr/projects/kindlm/packages/cloud/src/index.ts` and already handles both `GET /health` and the root path.

### Task 2 — MANUAL CHECKPOINT: Set up UptimeRobot monitoring

**This step cannot be automated — it requires creating an account on an external service.**

#### Steps

1. Go to [https://uptimerobot.com](https://uptimerobot.com) and sign up or log in
2. Click **Add New Monitor**
3. Configure:
   - **Monitor Type:** HTTP(s)
   - **Friendly Name:** `KindLM API Health`
   - **URL:** `https://api.kindlm.com/health`
   - **Monitoring Interval:** 5 minutes
   - **Monitor Timeout:** 30 seconds
   - **Alert contacts:** Add your team email address
4. Under **Advanced Settings**:
   - **Keyword Monitoring** (optional but recommended): Check for keyword `"status":"ok"` in the response body to catch degraded states that still return 200
5. Click **Create Monitor**

#### Alert Configuration

- Enable email alerts for **Down** and **Up** events
- Optionally add a Slack/Discord webhook for faster notification

#### Verify Setup

Once created, UptimeRobot will show a green status badge within 5 minutes if the endpoint is healthy. The free tier supports 50 monitors at 5-minute intervals — sufficient for this use case.

**Alternative services** (if UptimeRobot is not preferred):
- Better Uptime: https://betteruptime.com (1-minute intervals on free tier)
- Freshping: https://freshping.io (1-minute intervals, 50 checks free)

### Task 3 — Run the E2E smoke test against production

The smoke test at `packages/cli/tests/e2e/smoke.test.ts` exercises the full `init → test → upload` flow.

```bash
# From repo root — run smoke test pointing at production API
KINDLM_API_URL=https://api.kindlm.com \
  npm run test --workspace=packages/cli -- --testPathPattern=smoke
```

If the smoke test requires a valid API key, set it:

```bash
KINDLM_API_URL=https://api.kindlm.com \
KINDLM_API_KEY=<your-production-key> \
  npm run test --workspace=packages/cli -- --testPathPattern=smoke
```

**Expected outcome:** All assertions pass, exit code 0.

Check the smoke test file at `/Users/petr/projects/kindlm/packages/cli/tests/e2e/smoke.test.ts` to understand what fixtures it uses and what assertions it makes before running.

### Task 4 — Full E2E manual verification (success criterion 4)

Perform the full flow manually using the published CLI:

```bash
# Install the published CLI globally
npm install -g @kindlm/cli@1.0.0

# Verify version
kindlm --version
# Expected: 1.0.0

# Run full flow
kindlm init
kindlm test
kindlm upload
```

Then navigate to the dashboard and confirm results are visible.

**Checklist:**
- [ ] `kindlm --version` returns `1.0.0`
- [ ] `kindlm init` completes without error
- [ ] `kindlm test` runs and produces output
- [ ] `kindlm upload` uploads results successfully
- [ ] Dashboard shows the uploaded results
