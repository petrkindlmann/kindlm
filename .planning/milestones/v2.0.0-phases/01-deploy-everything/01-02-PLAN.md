
3. Deploy to production (default env in wrangler.toml):
   ```bash
   cd packages/cloud && npx wrangler deploy
   ```
   Expected output includes: `Published kindlm-api` or similar success message.

4. Verify deployment:
   ```bash
   curl -s -o /dev/null -w "%{http_code}" https://api.kindlm.com/health
   ```
   Expected: `200`

**OPS-03: Run D1 migrations on production**

5. Apply all pending D1 migrations to production database `kindlm-prod`:
   ```bash
   cd packages/cloud && npx wrangler d1 migrations apply kindlm-prod --remote
   ```
   This will show which migrations are pending and apply them. All 13 migrations (0001-0013) should be either already applied or newly applied.

6. Verify migrations applied:
   ```bash
   cd packages/cloud && npx wrangler d1 migrations list kindlm-prod --remote
   ```
   Expected: All 13 migrations show as applied.

**Important ordering note:** Deploy the Worker BEFORE running migrations. The Worker code must be updated to handle the new schema. However, all migrations are backward-compatible (they add columns/tables, don't remove), so the order is safe either way. Deploy first is preferred because the new code expects the new schema.

Actually, the safer order for D1 is: migrations first (add new tables/columns), then deploy (new code that uses them). The old Worker code won't break from new columns existing. Run migrations first, then deploy.

**Revised order:**
1. Wrangler auth check
2. Build
3. D1 migrations apply (OPS-03)
4. Wrangler deploy (OPS-04)
5. Health check
