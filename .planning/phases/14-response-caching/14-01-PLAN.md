    - deepSortKeys({b:1,a:2}) returns {a:2,b:1}
    - deepSortKeys({b:{d:1,c:2},a:3}) returns {a:3,b:{c:2,d:1}}
    - deepSortKeys([{b:1,a:2}]) returns [{a:2,b:1}]
    - deepSortKeys(null) returns null, deepSortKeys(42) returns 42
    - computeCacheKey produces identical hash for {a:1,b:2} and {b:2,a:1} in request params
    - readCacheEntry returns null for entries older than TTL (default 86400000ms)
    - readCacheEntry returns entry for entries within TTL
    - readCacheEntry respects cacheTtlMs from .kindlm/config.json when present
    - caching-adapter does NOT call writeCacheEntry when response.finishReason === "error"
    - caching-adapter does NOT call writeCacheEntry when response.text === "" AND response.toolCalls.length === 0
    - caching-adapter DOES call writeCacheEntry for successful responses with text
    - caching-adapter DOES call writeCacheEntry for successful responses with toolCalls
    - caching-adapter sets fromCache: true on cache hits
    - caching-adapter does NOT set fromCache on cache misses (live responses)
  </behavior>
  <action>
    **In cache.ts (per D-01, D-02, D-11, D-12, D-13):**

    1. Add `deepSortKeys(value: unknown): unknown` pure helper function:
       - If value is an array, map each element through deepSortKeys
       - If value is a non-null object, sort keys alphabetically and recurse on values
       - Otherwise return value unchanged

    2. Update `computeCacheKey`: replace `JSON.stringify(payload)` with `JSON.stringify(deepSortKeys(payload))`

    3. Add `getCacheTtlMs(): number` function:
       - Try to read `.kindlm/config.json` from `process.cwd()`
       - Parse JSON, look for numeric `cacheTtlMs` property
       - Return it if valid positive number, otherwise return 86_400_000 (24 hours)
       - Catch all errors and return default

    4. Update `readCacheEntry`: after parsing the entry and before returning it, compute age as `Date.now() - new Date(parsed.cachedAt).getTime()`. If age > getCacheTtlMs(), return null (per D-13: do NOT delete the stale file).

    5. Fix Windows path bug in `writeCacheEntry`: replace `filePath.slice(0, filePath.lastIndexOf("/"))` with `dirname(filePath)` from `node:path` (already imported as `join`; add `dirname` to the import).

    6. Export `deepSortKeys` for testing.

    **In caching-adapter.ts (per D-03, D-04, D-08):**

    1. In the `complete()` method, after `const response = await inner.complete(request)`:
       - Add guard before writeCacheEntry:
         ```
         const isCacheable = response.finishReason !== "error" &&
           (response.text !== "" || response.toolCalls.length > 0);
         ```
       - Only call `writeCacheEntry(key, response)` when `isCacheable` is true

    2. On cache hit return, add `fromCache: true`:
       ```
       return { ...cached.response, latencyMs: 0, fromCache: true };
       ```
       Note: This requires `fromCache` to exist on ProviderResponse — Task 2 adds it to the type. The spread will work at runtime regardless, and TypeScript will see it once Task 2 completes. If building in order, add the type first.

Test readCacheEntry TTL expiry using vi.useFakeTimers or by writing entries with old cachedAt timestamps. Test getCacheTtlMs with mock config.json.

    - `caching-adapter.test.ts`: Create a mock ProviderAdapter, wrap with createCachingAdapter. Test that error responses are not cached (call complete twice with error response — second call should still hit the real adapter). Test that empty responses (no text, no toolCalls) are not cached. Test that successful responses ARE cached (second call returns fromCache: true). Test that fromCache is true on cache hits.
